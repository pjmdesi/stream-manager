# Full-App Bug Sweep (2026-07-03)

Method: 12 parallel code reviewers, one per subsystem, over the whole `src` tree (~48k lines), findings deduplicated and the serious ones re-verified against the source by hand. Everything here is code-derived: each item has a concrete way to confirm it in the running app.

Legend:
- **(verified)**: I read the cited code myself and the defect is real as described.
- **(reported)**: reviewer finding with precise citations; spot checks of that reviewer's other claims passed, but treat as "very likely" not "certain".
- 🔴 data loss or corruption · 🟠 wrong behavior in normal use · 🟡 stuck states, edge cases · ⚪ minor

Mark items off like the release checklist: `[x]` fixed or confirmed-not-a-bug, `⚠` + note if flaky.

## 1. Data loss & corruption (fix before anything else)

- [x] 🔴 **_meta.json can silently reset to empty, then the loss is made permanent** (verified): `readAllMeta` returns `{}` on ANY read failure (torn write, file briefly locked by Synology Drive, EBUSY) with no error surfaced (`src/main/ipc/streams.ts:339-345`), and `writeAllMeta` is a plain truncate-write with no temp-file+rename and no backup (`:347-366`). After a bad read, the next auto-write (videoMap refresh stub write, any meta save) overwrites the still-recoverable file with near-empty data: every stream's games, comments, YT links, archived flags gone. Recommend: atomic write (tmp + rename), refuse to write when the last read failed, keep a rolling `.bak`. This is the single most important item in this file since `_meta.json` is the whole database.
- [x] 🔴 **Background videoMap refresh writes a stale whole-file snapshot minutes later** (verified): `streams:list` fires `refreshVideoMaps(...)` unawaited and its `.then` calls `writeAllMeta(dir, allMeta)` with the snapshot captured at scan time (`src/main/ipc/streams.ts:640, 844-857`). Any meta save landing while probes run (first scan of a library, new recording arriving) is clobbered and the UI reverts on the follow-up `streams:changed`; deleted streams can resurrect. Same pattern with a smaller window in `src/main/ipc/converter.ts:778-791`.
- [x] 🔴 **Archive replace-original can delete BOTH the original and the finished conversion** (verified): `handleComplete` runs `unlinkSync(input)` then `renameSync(output, final)`; the catch deletes the output, and `handleError` retry-deletes it again (`src/main/ipc/converter.ts:636-650, 674-676`). A file handle held by Synology/AV at the wrong moment (rename fails after unlink succeeded) permanently loses both copies (unlink skips the recycle bin). Safe order: rename original aside, rename temp in, then delete the backup.
- [x] 🔴 **Re-combining a stream destroys the previous combined file** (verified): default output is always `<folder> combined.mkv` (`src/renderer/src/components/pages/CombinePage.tsx:37-42`) and `combine:run` passes `-y` (`src/main/ipc/combine.ts:37-45`), so a second combine truncates the first, and Combine All even feeds the old combined file in as an input. With "Delete source files" checked, sources are `unlinkSync`'d permanently on exit 0 with zero output verification and no codec/resolution compatibility check before the `-c copy` concat (reported).
- [x] 🔴 **Dump-mode stream delete destroys metadata first, then swallows trash failures** (verified): `streams:deleteStreamFiles` deletes the meta entry and writes `_meta.json` BEFORE trashing, each trash wrapped in `catch {}` (`src/main/ipc/streams.ts:1219-1226`). A file locked by an external app leaves the files on disk but the metadata already gone, silently. Folder mode does it in the correct order.
- [x] 🔴 **Thumbnail editor: "Close session" discards up to 500ms of edits, every time** (reported, mechanism verified): closing only sets `setMode('overview')`, never flushes `autoSaveTimer`; the pending save then bails on the unmounted stage (`src/renderer/src/components/pages/ThumbnailPage.tsx:4038-4046, 2790`). A burst of edits re-arms the debounce each time, so the tail of the burst is lost. Dirty template sessions are discarded entirely with no prompt.
- [x] 🔴 **Thumbnail autosave landing after a stream switch writes stream B's pixels as stream A's PNG** (reported): `openStreamEditor` doesn't cancel pending/in-flight saves; the capture happens after awaits that can take up to 5s with a broken image layer (`ThumbnailPage.tsx:3233-3325, 2797-2799`). Result is a persistent JSON/PNG pair desync.
- [x] 🟠 **Opening a thumbnail from the Recent list drops stream meta, then bakes blank merge fields into the PNG** (verified): `openFromRecent` passes no `meta`/`totalEpisodes` (`ThumbnailPage.tsx:3327-3329`), so `{title}`/`{game}` render as empty strings; any edit (or the missing-PNG self-heal) re-exports the thumbnail with the text gone.
- [x] 🟡 **Corrupt/missing variant JSON reroutes through the template picker and overwrites variant 1** (reported): `confirmPickTemplate` hardcodes ordinal 1 and eagerly saves (`ThumbnailPage.tsx:3357, 3372-3375`), so opening broken variant 2 and picking a template clobbers healthy variant 1.
